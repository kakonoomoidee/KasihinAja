# v1.4.0 - Phantom Navy

**Release Date**: 2026-04-01
**Codename**: Phantom Navy

---

## Overview

v1.4.0 is a major sprint release delivering dynamic media pricing, a full database schema migration to native JSON, OBS censorship for banned donors, a global UI overhaul to a premium Dark Navy Glass aesthetic, and native TikTok media share support inside OBS alerts.

---

## Changes

### Database

- **JSON Column Migration**: Converted `streamer_profiles.custom_blacklist` from `TEXT` (storing a JSON string) to a native `JSON` column (`DataTypes.JSON`, `defaultValue: []`). All backend read and write paths now operate on plain JavaScript arrays with no manual `JSON.parse` or `JSON.stringify`.
- **`banned_keys` Column**: Added `streamer_profiles.banned_keys` as a native `JSON` column (`allowNull: false`, `defaultValue: []`). Stores an array of checksummed donor wallet addresses permanently excluded from a streamer's broadcast.
- **Media Timing Persistence**: Added `donation_history.media_start` (`INTEGER`, `defaultValue: 0`) and `donation_history.media_duration` (`INTEGER`, `defaultValue: 5`). These fields persist the exact YouTube start time and computed playback duration from the intent JWT, ensuring replays from the history tab broadcast the identical timing state as the original live alert.

---

### Backend

- **Dynamic Media Duration Calculation**: `intentController.js` now fetches the target streamer's `media_price_per_second` (`FLOAT`, default `0.0005`) and `vn_fixed_price` (`FLOAT`, default `0.01`) from the profile. For YouTube and TikTok media, playback duration is calculated as `floor(amount / pricePerSec)`. A strict 30-minute cap (`Math.min(1800, rawDuration)`) is enforced server-side. Voice Notes are granted a fixed 30-second duration at or above the `vn_fixed_price` threshold. Requests that cannot satisfy the minimum 1-second duration are rejected with a `400` error.
- **`youtube_start` Persistence**: `intentController.js` extracts `youtube_start` from both the top-level request body and `media_data`, packing it explicitly into the JWT `media_data` payload. `evmListener.js` maps this to `media_start` in `DonationHistory.create` and in the `VERIFIED_DONATION` broadcast payload.
- **OBS Censorship Middleware**: `evmListener.js` now checks `profile.banned_keys` before constructing the broadcast payload. If the donor's lowercase wallet address is found in the array, the OBS payload is sanitized: `donor_name` is overridden to `"BANNED USER"`, `message` is cleared to `""`, and all media fields (`media_type`, `media_url`, `vn_data`, `media_start`, `media_duration`, and the full `media_data` sub-object) are zeroed out. The `amount` field is preserved so the donation value remains visible on stream. The raw database record is saved with the original data; the history API already excludes banned donors via `Op.notIn`.
- **Ban Endpoint**: New `POST /api/profile/:address/ban` endpoint (`banDonor` in `profileController.js`). Accepts `{ donor_address }` in the request body. Appends the lowercased address to `banned_keys`, calls `profile.changed("banned_keys", true)` to force Sequelize mutation detection on the JSON column, and returns the updated `banned_keys` array.
- **History Ban Filter**: `historyController.js` `getHistory` now fetches the streamer profile, reads `banned_keys` as a native array, and applies `donor_address: { [Op.notIn]: bannedKeys }` to the `DonationHistory.findAll` query. Banned donor records are excluded from the dashboard history view without deleting the underlying data.
- **Streamer Pricing Configuration**: `profileController.js` `updateProfile` now accepts and persists `media_price_per_second` and `vn_fixed_price` from the signed payload. Both fields are included in `findOrCreate` defaults and in the conditional update block.

---

### Frontend - UI Overhaul

- **Dark Navy Glass Design System**: Executed a global visual overhaul across `Dashboard.jsx`, `PublicProfile.jsx`, `AlertOverlay.jsx`, `MilestoneOverlay.jsx`, and all modular dashboard components. All gradient-based backgrounds, `bg-gradient-to-*`, `from-*`, `to-*`, and `via-*` Tailwind classes were removed. The application now uses a monochromatic deep navy base (`#0f172a`) with a single atmospheric radial glow (sky-cyan, 5% opacity) defined in `index.css`. All containers and cards use the `bg-white/[0.03] backdrop-blur-xl border border-white/[0.08] rounded-2xl` glass system.
- **Dashboard Modularization**: `Dashboard.jsx` was split into focused component files under `src/components/dashboard/`: `OverviewTab.jsx`, `HistoryTab.jsx`, `PageSetupTab.jsx`, `AlertsTab.jsx`, `ModerationTab.jsx`, and shared subcomponents.

---

### Frontend - UX Fixes

- **Invisible Modal Fix (HistoryTab)**: The "Replay" and "Ban" confirmation modals in `HistoryTab.jsx` were invisible because the parent container's `fade-in` CSS animation applied a `transform`, creating a new containing block that broke `position: fixed` modals subtrended inside it. Fixed by wrapping all modal mounts in `ReactDOM.createPortal(..., document.body)`, escaping the stacking context entirely.
- **Duration Estimation UI (PublicProfile)**: Added a "Media Info" panel below the amount input on the public tipping form. The panel appears when YouTube or TikTok is selected and `media_price_per_second > 0`. It displays the current rate in ETH/sec, the 30-minute cap, and a live estimated duration (`Math.min(1800, floor(amount / pricePerSec))`) that updates as the donor types.
- **Media Pricing Configuration (PageSetupTab)**: Added a "Media Pricing" settings card to the streamer dashboard allowing streamers to configure their `media_price_per_second` (ETH/sec for video media) and `vn_fixed_price` (fixed ETH amount for Voice Notes), with helper text showing defaults and the duration formula.
- **Voice Note 30-Second Limit**: `PublicProfile.jsx` now enforces a hard client-side 30-second recording limit. A `vnAutoStopRef` setTimeout fires `stopRecording()` automatically at 30 seconds. The timer is cleared immediately if the user stops recording manually, preventing a double-stop race condition.
- **Ban Action (Dashboard)**: The `banKey` function in `Dashboard.jsx` is now async. It calls `POST /api/profile/:address/ban`, updates local `bannedKeys` state from the server response, and immediately removes the banned donor's cards from the `history` state array using `.filter()` — no page reload required.

---

### OBS Alert Overlay

- **TikTok Native Video Support**: `AlertOverlay.jsx` now supports TikTok media share. The overlay uses a direct MP4 proxy URL (`tikwm.com`) to render a native `<video>` element with autoplay and loop. A fallback error state is displayed if the proxy fetch fails. `TikTokInput.jsx` in the donor form was also updated to use the same native video player with custom play/mute controls and a hover-revealed control bar.
- **YouTube Looping Fix**: The YouTube iframe `src` now includes `&loop=1&playlist={videoId}`. YouTube ignores `loop=1` in embedded iframes without the `playlist` parameter set to the same video ID.
- **Strict Duration Enforcement**: `processQueueRef` in `AlertOverlay.jsx` now reads `next.media_duration` at the root level of the broadcast payload (the DB-persisted value) with fallback to `next.media_data?.duration`. `displayTime = durationSec * 1000` with no heuristic branching. The dismiss timer fires at exactly the backend-computed duration.
- **`youtube_start` Fix**: `AlertCard` reads `alert.media_start` first (root-level DB field) via null-coalescing, with fallback through `media_data.youtube_start` and `alert.youtube_start`. This ensures the correct start time is used for both live alerts and replays.

## Migration Notes

Run the following migration scripts before deploying the backend:

```
node Backend/migrations/migrate_v1_5.js
node Backend/migrations/migrate_v1_6.js
node Backend/migrations/migrate_v1_7.js
```

`migrate_v1_7.js` performs a raw SQL `UPDATE` to normalize any existing `NULL` or empty `TEXT` values in `custom_blacklist` and `banned_keys` to `'[]'` before altering the column types to `JSON`. This prevents MySQL data truncation errors on `changeColumn`.
